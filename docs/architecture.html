Greenhouse Platform Architecture

This page provides a concise reference view of the Greenhouse platform across user experiences, integration services, identity, observability, and Azure-aligned governance practices.

Azure Well-Architected Framework Azure landing zones

Logical architecture

The Greenhouse platform is organized into a layered model so client experiences, integration endpoints, shared platform services, and governance controls remain clearly separated.

Business intent

Support greenhouse operations with secure application experiences, partner integrations, and a platform foundation that can scale across environments without duplicating control-plane services.

Priority capabilities

  • Centralized authentication and workload identity with Microsoft Entra ID.
  • Managed API exposure and policy enforcement through Azure API Management.
  • Shared telemetry, diagnostics, and alerting with Azure Monitor.
  • Governed deployment patterns aligned to landing-zone architecture.

Core capability areas

Identity and access

Use Microsoft Entra ID for user sign-in, application identity, RBAC, and conditional access controls.

API and integration

Use Azure API Management to publish, secure, throttle, and observe partner and internal APIs consistently.

Operations and monitoring

Use Azure Monitor and diagnostic pipelines to baseline health, detect drift, and support incident response.

Platform governance

Separate foundational services from workload services and apply enterprise guardrails through landing zones.

Recommended implementation principles

  • Keep identity, networking, and monitoring standards centralized where possible.
  • Expose integrations through managed gateways instead of direct backend coupling.
  • Design for operational visibility from the start, not as a later add-on.
  • Use reference architectures to guide consistency across delivery teams.
